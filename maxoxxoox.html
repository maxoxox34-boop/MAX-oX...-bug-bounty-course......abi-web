CASE NO. 2026-0714 STATUS: OPEN CLASSIFICATION: PUBLIC SUBJECT: Web Application Security FILED BY: Max oX

Hunt bugs like an investigator, not a vandal. Document everything.

Bug bounty hunting is applied skepticism: you form a hypothesis about where a system's trust breaks down, test it inside permission, and write up what you find so someone can fix it. This manual walks the whole case — from first recon to a report a triage team actually wants to read.

Authorized Use Only

Before you open this file

Everything here is conceptual and defensive-education material. It will not teach you to attack a system you don't have permission to test — and you shouldn't. Unauthorized access is a crime in most jurisdictions, full stop.

01 — Getting Started

The roadmap, in order

This is one of the few places sequence actually matters — skipping steps to chase your first bounty is the most common way people burn out or get themselves banned from a platform.

01

Learn the terrain

Networking basics, HTTP end to end, how browsers and servers negotiate trust, comfortable Linux command line, and one scripting language (Python is the common choice). You can't spot an abnormality if you don't know normal.

02

Build a legal lab

Practice exclusively on environments built for this: PortSwigger Web Security Academy, OWASP Juice Shop, DVWA, or platforms like Hack The Box and TryHackMe. This is where you make mistakes safely.

03

Learn a methodology cold

Work through the OWASP Testing Guide and OWASP Top 10 until the categories are second nature. A methodology keeps you systematic instead of randomly poking at things.

04

Read the policy before you read the app

Pick a program with a clear, permissive scope on a platform like HackerOne or Bugcrowd. The scope and rules document is the contract that makes your testing legal — treat it like one.

05

Write your first report as a case file

Reproduction steps, impact, evidence, and a suggested fix — in that order. A report a developer can act on in five minutes gets triaged faster and paid faster than a vague one.

06

Build your playbook

Keep notes on what worked, what didn't, and which bug classes you're fastest at spotting. Specialization compounds — most experienced hunters have a "signature" vulnerability class.

02 — Methodology

How a case actually gets worked

Every engagement, big or small, moves through the same five phases. Rushing to phase three without finishing one and two is the most common rookie mistake.

Phase — Scoping

Reconnaissance & scoping

Read the program policy twice. Define the exact boundary of what's in scope. Passive recon only at this stage — public subdomain data, technology fingerprinting, certificate transparency logs. Nothing that touches the target directly yet.

Phase — Mapping

Mapping the attack surface

Crawl the application like a user would, then like an admin would. Catalog every endpoint, parameter, and role boundary. Note anywhere the app trusts client-supplied data to make a server-side decision.

Phase — Testing

Hypothesis-driven testing

For each input, ask: what trust boundary does this cross? Test that specific hypothesis, minimally and within scope — never on data or accounts that aren't yours to touch.

Phase — Verification

Verification & evidence

Reproduce reliably before reporting. Capture only the evidence needed to prove impact — a screenshot or log line, not a full data dump. Minimizing what you touch is part of doing this responsibly.

Phase — Reporting

Responsible reporting

Write the case file: clear title, steps to reproduce, business impact in plain language, and a remediation suggestion. Submit through the program's channel and hold disclosure until they've had time to triage and patch — that's the entire deal that makes bug bounty legal and welcome.

03 — Reference

Case files: common vulnerability classes

Conceptual summaries only — modus operandi, typical impact, and how it's normally prevented. Use these to know what to look for, not as attack code.

Critical

Cross-Site Scripting (XSS)

MOUntrusted input reaches the page without proper encoding, so the browser treats it as executable script instead of text.
Typical impactSession hijacking, UI manipulation, credential theft.
Usually prevented byContext-aware output encoding and a strict Content-Security-Policy.
Critical

SQL Injection

MOUnsanitized input reaches a database query and alters its logic instead of just its data.
Typical impactData exposure, tampering, or full database compromise.
Usually prevented byParameterized queries / prepared statements, never string-built SQL.
Critical

SSRF

MOA server is tricked into making a request to a destination the developer never intended — often an internal service.
Typical impactInternal network access, cloud credential theft.
Usually prevented byStrict allow-lists and network segmentation for outbound requests.
High

IDOR

MOAn object reference (an ID in a URL or request) is swapped, and the server never rechecks whether the requester actually owns that object.
Typical impactHorizontal privilege escalation, data leakage across accounts.
Usually prevented byServer-side authorization checks on every object access, not just at login.
High

Broken Authentication

MOWeaknesses in login, password reset, or session handling let an attacker impersonate a legitimate user.
Typical impactFull account takeover.
Usually prevented byMFA, secure session tokens, and rate-limited auth flows.
Medium

Business Logic Flaws

MOThe app's intended workflow is used out of order or with unexpected quantities in a way its logic never accounted for.
Typical impactFinancial fraud, discount abuse, workflow bypass.
Usually prevented byServer-side validation of every state transition, never trusting client-side flow.

05 — Non-Negotiable

Rules of engagement

  1. Get explicit authorization first. A program's published scope is your permission slip — testing anything outside it is unauthorized access, even if you found it by accident.
  2. Touch the minimum. Prove impact with the smallest possible action. Don't download data, don't pivot further than needed to demonstrate the bug.
  3. Never test third parties. If a target's infrastructure touches a vendor or partner system, that system is out of scope unless explicitly named.
  4. No denial-of-service testing. Don't run anything that could degrade availability for real users, ever, even "just to check."
  5. Follow the disclosure timeline. Report privately through the program's channel and hold public disclosure until the team has confirmed a fix or given explicit permission.
  6. Social engineering and physical testing require separate, explicit authorization — a web bug bounty scope almost never covers them by default.
Unauthorized access to computer systems is a criminal offense in most countries — e.g. the Computer Fraud and Abuse Act (US), the Computer Misuse Act (UK), and equivalent laws elsewhere. A bug bounty program's scope is what converts testing from a crime into legitimate, often paid, security research. When scope is unclear, ask the program — don't assume.
